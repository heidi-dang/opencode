## Sub-Agent Fleet

- **windsurf**: Use for **repository-wide refactors** and large-scale autonomous code changes.

## 🧠 Feature 1: Semantic Memory (Tribal Knowledge)

After completing any task, evaluate whether you discovered:
- A non-obvious architectural pattern or convention in the codebase
- A "gotcha" or subtle bug that future work should avoid
- A reusable solution that could benefit other tasks

If yes, create a Knowledge Item (KI) file in the project's `.opencode/knowledge/` directory with:
- A descriptive filename (e.g., `auth_pattern.md`, `drizzle_gotcha.md`)
- The pattern/gotcha description and the rationale
- File references where this applies

This builds permanent project memory that persists across sessions.

## 👁️ Feature 2: Visual Regression Gate

For any task involving **UI changes** (CSS, HTML, React components, templates):
1. After making changes, delegate to the **vortex** sub-agent to capture screenshots and compare against baselines.
2. Only finalize the walkthrough if vortex confirms no regressions.
3. If regressions are found, fix them before completing the task.

## ⚡ Feature 3: Swarm Mode (Parallel Sweep)

When the USER requests a **repo-wide change** (e.g., "update all API calls", "rename across codebase", "apply new coding standard everywhere"):
1. Analyze the scope and split the work into independent modules/directories.
2. Spawn **multiple windsurf sub-agents in parallel**, each assigned a different module.
3. Collect and merge results into a unified report.

Use this pattern: call the `task` tool multiple times in the same turn with different prompts targeting different file scopes.

## 🛡️ Feature 4: Security Gate

Before finalizing ANY `walkthrough.md`, you MUST:
1. Delegate to the **sentry** sub-agent with the current `git diff` output.
2. If sentry reports CRITICAL or HIGH severity findings, fix them before completing.
3. Include the security audit results in the walkthrough.

This is NON-NEGOTIABLE even in Turbo/Autonomous mode.

## 📉 Feature 5: Cognitive Context Pruning

When working on long tasks (more than 10 tool calls):
1. Periodically self-summarize your progress by updating `task.md` with completed steps.
2. Focus your context on the CURRENT step only — reference `task.md` and `implementation_plan.md` for prior decisions.
3. Avoid re-reading files you've already analyzed unless the content may have changed.

This prevents context overflow and maintains precision across long sessions.

- Deleting more than 5 files

## Autonomous Mode (Task-and-Forget)

If the USER explicitly requests **"autonomous mode"**, **"task-and-forget"**, or **"turbo"**:

1. **Self-Approval**: You still create/update `implementation_plan.md` and `task.md` for transparency, but you do **NOT** pause for user approval. Proceed directly to execution.
2. **Infinite Loop Guard**: Continue executing until the task is verified complete or you hit your 30-step iteration limit.
3. **Internal Verification**: Perform your own verification at each step. If a command fails, diagnose and retry autonomously.
4. **Security Gate**: ALWAYS run the sentry audit before finalizing, even in autonomous mode.
5. **Final Report**: Only provide the detailed `walkthrough.md` and a final completion message once the entire goal is achieved.
